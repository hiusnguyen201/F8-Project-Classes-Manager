
# Codebase Overview

This document provides a high-level overview of the Classes Manager codebase, its structure, and key components.

## Directory Structure

The project follows a logical structure, with the majority of the application code residing in the `src/` directory.

```
.
├── public/                 # Static assets (CSS, JS, images, uploads)
├── src/
│   ├── app.js              # Main application entry point
│   ├── constants/          # Application-wide constants
│   ├── database/           # Database configurations and migrations
│   ├── helpers/            # Helper functions and utilities (e.g., passports)
│   ├── http/               # Core HTTP request handling
│   │   ├── controllers/    # Request handlers (web and potentially API)
│   │   ├── middlewares/    # Request middleware (authentication, validation, etc.)
│   │   └── services/       # Business logic services
│   ├── models/             # Database models (Sequelize)
│   ├── resources/
│   │   └── views/          # Frontend view templates (EJS)
│   │       ├── admin/      # Admin-specific views
│   │       ├── auth/       # Authentication views
│   │       ├── errors/     # Error page templates
│   │       ├── layouts/    # Layout templates
│   │       └── mail/       # Email templates
│   ├── routes/             # Application routes (web and API)
│   └── utils/              # Utility functions
├── openwiki/               # Documentation generated by OpenWiki
├── .env.example            # Example environment variables
├── package-lock.json
├── package.json            # Project dependencies and scripts
└── README.md               # Project description (if exists)
```

For technology stack and core domains, see [Architecture Overview](architecture.md) — this page focuses on directory layout and workflow.

## Development Workflow

The typical development workflow involves:

1.  **Setup**: Cloning the repository and setting up the development environment (installing dependencies, configuring the database).
2.  **Development**: Making code changes in `src/` according to feature requirements or bug fixes.
3.  **Testing**: Running relevant tests to ensure code quality and functionality.
4.  **Committing**: Committing code changes with descriptive messages, adhering to project conventions.
5.  **Building/Deployment**: Preparing the application for deployment.

This overview should provide a foundational understanding of the project's structure and the technologies it employs.
